# docs/v0.1-soak-checklist.md — v0.1 Internal Soak 手动验收清单

> 本清单用于 v0.1 internal soak 前的手动验收。
> 所有项均为手动操作验证，不替代单测。测试环境：本地 backend + Emerald-client。
> 每项标注 `[ ]` 待验证 / `[x]` 已通过 / `[~]` 已知限制/跳过。

---

## 1. 启动验收

### 1.1 Backend 启动

- [ ] `python main.py` 启动，无异常退出，日志无 ERROR 级别错误
- [ ] NapCat 连接成功（或 `qq.enabled=false` 时日志确认跳过 QQ 初始化）
- [ ] 管理面板 `http://127.0.0.1:8080` 可访问，Bearer token 生效
- [ ] 调度器 loop 启动，日志出现 scheduler tick 行
- [ ] `GET /garden/state` 返回合法 JSON（非 500）

### 1.2 Emerald-client 启动

- [ ] `npm run tauri dev`（或打包版）启动，窗口正常显示
- [ ] WebSocket `/ws/desktop` 连接成功，日志出现 `DesktopChannel active`
- [ ] 聊天面板加载，历史记录显示正常（无乱码、无 null 占位）
- [ ] 角色卡/mood 区域显示当前活跃角色，无报错

---

## 2. 基础聊天无双显

### 2.1 普通 chat（桌面通道）

- [ ] 在聊天框发送一条消息，他回复**恰好一条**，不重复出现
- [ ] WS 协议：`channel_message` + `message_segments` 各出现一次，`msg_id` 相同
- [ ] `ChatPanel` 按 `msg_id` 合并，屏幕上只渲染一条回复

### 2.2 Desktop wake（桌宠唤醒）

- [ ] 触发 desktop wake（最小化后恢复/移动窗口等方式），他**只发一条**主动问候
- [ ] 检查 server 日志：`perceive_event` 只出现一次 `ACCEPTED`（无重复 `ACCEPTED`）
- [ ] 第二次连续触发 wake，90 秒内被 `DUPLICATE` 拦截，他不再发话

### 2.3 HTTP send + WS 不双显

- [ ] `POST /desktop/chat` 发送消息，HTTP response 返回回复
- [ ] WS 上只收到一份 `channel_message`（`turn_sink fanout` 不向发起通道重复推送）
- [ ] `POST /upload/ingest` 上传图片，他回复恰好一条，不重复

---

## 3. Dream 验收

### 3.1 进入梦境

- [ ] 点击入梦按钮（`POST /dream/enter`），`dream_state.status` 变为 `DREAM_ACTIVE`
- [ ] 入梦后 `/desktop/chat` 和 `/mobile/chat` 被后端拒绝（返回非 200 或空结果），不进现实 pipeline
- [ ] QQ 消息在 dream active 期间被 `perceive_event` 的 Dream Guard 拦截（`BLOCKED_DREAM`）

### 3.2 梦内聊天

- [ ] `POST /dream/chat` 发送消息，收到梦境回复，不写现实 short_term/memory
- [ ] 梦内回复在 Dream 聊天面板显示，不出现在现实聊天面板
- [ ] `GET /dream/state` 返回正常（含 `body_state / emotional_tension`）

### 3.3 退出梦境

- [ ] 点击 WAKE / Esc 触发 `POST /dream/exit`，`dream_state.status` 变回 `REALITY_CHAT`
- [ ] 退梦后 `/desktop/chat` 恢复正常，他可正常回复
- [ ] 梦境原文写入 `data/runtime/dreams/{char_id}/archive/`（sandbox 模式）或正常路径

---

## 4. Dream 与 Reality 隔离

### 4.1 Reality trigger 不进入 Dream UI

- [ ] 梦境 active 期间，调度器定时触发器被 Dream Guard 拦截，不向 Dream 聊天面板推送 reality 内容
- [ ] Emerald-client Dream 聊天窗不接收现实 WS `channel_message`（Dream UI 使用独立 dream API，不订阅 `/ws/desktop` reality 流）

### 4.2 Trigger 不写 short_term

- [ ] 触发一个 scheduler stimulus（如碎碎念/早安），查看 `data/history/{uid}.json`：stimulus 事件本身无独立写入记录；只有助手回复（`assistant` role）由 `record_assistant_turn` 写入
- [ ] Trigger audit log（`data/logs/trigger_audit.jsonl` 或等效）出现对应 `gate_result=ACCEPTED` 行，含 `event_id / dedupe_key / source / char_id`

---

## 5. 多角色切换不串数据

验收前提：系统中已注册 `yexuan`、`character_b`、`J-5412` 三个角色（或按当前实际注册数量调整）。

### 5.1 切换后 mood/activity/observations 独立

- [ ] 切换到角色 A，查看 `GET /mood/state`，返回 A 的 mood；切换到角色 B，返回 B 的 mood，两者不混用
- [ ] 切换角色后，聊天历史（short_term）显示对应角色的历史，不出现跨角色内容
- [ ] 切换角色后，mid_term / episodic 召回内容属于对应角色，不出现另一个角色的记忆片段

### 5.2 Garden/sensor 写 active_character

- [ ] 浇水或 sensor 事件写入后，`data/garden/{char_id}/` 路径正确对应当前活跃角色
- [ ] 无 `yexuan` 硬编码回退：若当前活跃角色非 yexuan，路径仍正确

## 6. 数据安全

### 6.1 secrets / runtime / data 不进 git

- [ ] `git status` 显示 clean 或只有预期的代码文件
- [ ] `data/` 目录整体被 `.gitignore` 覆盖，无意外暂存
- [ ] `config.yaml`（含 API key / admin secret）不在 git 跟踪文件列表中
- [ ] `.env` / `*.key` / `*.pem` 文件（若存在）不被跟踪
- [ ] `data/runtime/`、`data/history/`、`data/logs/` 均无跟踪文件
