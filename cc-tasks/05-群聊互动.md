# CC-05 · 群聊：让角色互相接话，不再只盯着「我」（f）

> 纯【后端 qq-st-bot】。先读 `docs/stage.md` + `core/stage/arbiter.py` + `core/stage/runner.py`。
> **可与 CC-04 并行，但两者都碰 `core/stage/models.py`（StageSettings 默认值）——同时改会冲突。建议 04 先合，05 再开；或同一人顺序做。**

## 现状（已核对，这是关键）
- `runner.py` **已经有 AI 接话链**（`run_owner_turn` 第 121–141 行）：owner 说完后，循环 `ai_chain_depth < max_ai_chain_depth`（默认 **2**），每轮排除「上一个发言者」，让另一个角色回应最新消息。**所以"角色互相接话"的骨架是有的。**
- 但 `arbiter.score_candidates` 用 `transcript[-1]` 打分，权重项：`talkativeness*0.5` + `addressed(0.9)` + `keyword(≤0.6)` − `recency_penalty`。
  - `_addressed()` 只在「被 @ 或被点名」时给 0.9。**角色之间几乎不会 @ 对方**，所以当上一句是另一个角色说的时，候选人的 `addressed=0`，得分基本只剩 `talkativeness*0.5`（≈0.25）再减 recency。
  - runner 第 104 行有 `pick.total < respond_threshold` 的门槛——上面那点分数大概率**低于阈值，AI 链第一轮就断**，于是角色只回 owner、不接彼此。**这就是"只在意我"的机制根因（叠加人设）。**

## 目标
让角色在「另一个在场角色刚说了话」时也有合理的发言意愿，形成自然的多角色对话，同时**不失控**（保留 N/M 上限和链深上限防刷屏）。

## 实现

### 1. arbiter 加「回应同伴」正向项 `core/stage/arbiter.py`
在 `score_candidates` 的 `parts` 里新增一项：当 `transcript[-1].speaker_id` 是**另一个角色**（非 owner、非自己）时，给一个 base 意愿分。
```python
latest_speaker = transcript[-1].speaker_id if transcript else "owner"
peer_spoke = latest_speaker not in ("owner",) and latest_speaker != char_id

parts = {
    "talkativeness": talkativeness * 0.5,
    "addressed": 0.9 if char_id in addressed else 0.0,
    "topic": _keyword_relevance(stage, char_id, latest_text),
    "peer_reply": PEER_REPLY_BASE * talkativeness if peer_spoke else 0.0,  # 新增
    "recency_penalty": -_recency_penalty(char_id, transcript),
}
```
- `PEER_REPLY_BASE` 建议 0.35–0.45（模块常量，可调）。乘 talkativeness 让"话痨"角色更爱接话。
- 这样"别人刚说了、我想接一句"会有一个 base 分，足以越过 `respond_threshold`，但不至于盖过"被点名/被 @"。

### 2. 调链深与阈值（防断也防刷屏）`core/stage/models.py` + runner
- `StageSettings.max_ai_chain_depth` 默认 2 → 提到 **3**（让 A→B→A 这种来回成为可能）。
- 确认 `respond_threshold`（runner:104 用到，找它定义）默认值；若偏高导致 peer_reply 仍过不去，**给 AI 链轮次单独一个略低的阈值**（chain 比首轮更宽松），或在 runner 链循环里用 `respond_threshold * 0.8`。二选一，注释说明。
- **保留** `max_responders`（每轮回应人数上限）和 `recency_penalty`（刚说过的人降权，避免一个角色连说）——这两个是防刷屏的闸，不要动其方向。

### 3. prompt 侧：明确允许回应同伴 `core/stage/runner.py:generate_reply` / `core/stage/context.py`
- 确认每个角色的生成 prompt 里，**共享 transcript 带了其他角色的发言且标注了 speaker 名**（runner 把整个 transcript 传进 `generate_reply`，先核实 context/projection 是否保留 speaker 标签）。
- 在群聊场景指令里加一句框定（不硬编码台词）：「你在一个有其他角色在场的群聊里。你可以回应在场的其他角色，不必每句都只对用户说话；也可以接、反驳或补充别人刚说的话。」
- 注意多角色隔离契约：各角色仍只读自己该读的记忆，互相接话只基于**共享 transcript 的可见发言**，不串记忆。

### 4. （可选）让"被点名"也认中文名/别名
- `_addressed()` 已认 `@char_id` / `@name` / `name in text`。可顺手确认 `get_char_name` 的别名也覆盖，避免点名了却没认出来。

## 验收
- 构造一个 ≥3 人群，owner 抛一个开放话题 → 观察是否出现角色 A 说完、角色 B 自然接话（而不是各自只回 owner 后冷场）。
- 「被 @ 某角色」时仍优先该角色回应（addressed 权重没被 peer_reply 盖过）。
- 不出现单角色连续刷屏（recency_penalty + max_responders 仍生效）。
- 群里只有 owner 单独对话时行为不退化（peer_reply 在没有同伴发言时为 0）。
- 补/改 arbiter 单测：构造 transcript[-1] 为同伴发言，断言该候选人得分高于 respond_threshold；构造 owner 发言，断言行为不变。
