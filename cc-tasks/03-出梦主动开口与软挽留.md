## Part A · 验证「出梦后主动开口」为何从没触发（d-1）

### 现状（已核对）
- 触发器**已写且已注册**：`core/scheduler/triggers/dream_exit.py`（`propose()` + `_make_execute()`），`proposer_registry.py:66` 已 import `core.scheduler.triggers.dream_exit`。
- 触发条件：仅当 `dream_state.status == REALITY_AFTERGLOW`、afterglow 已就绪、且 `last_greeted_dream_id != 当前 dream_id`（一梦一次）。config 无 `scheduler.dream_exit_trigger` 键 → 默认 True（不是被关掉）。
- 出梦链路：`_do_close_dream()`（`dream_pipeline.py`）把 status 置 `REALITY_AFTERGLOW`，异步 `_generate_summary_bg()` → 完成后 `wire_afterglow_from_summary()` 写 `afterglow_residue.json`。

### 这不是开发任务，是定位任务。按序排查：
1. **跑一场真梦**（test 模式 `python run_test.py` 或正常模式）：进梦→聊 ≥4 轮→正常出梦（**走软出口或硬退都行**）。
2. 出梦后立刻查 `dream_state`：`status` 是否 `REALITY_AFTERGLOW`？`last_dream_id` / `char_id` 是否有值？（读对应 `data/runtime/.../dream_state` 文件，或加临时 log。）
3. 查 `afterglow_residue.json` 是否生成（summary bg 是否跑完）。**scenario/mirror 模式不写 afterglow**——确认你测的是 sandbox 模式。
4. 看 scheduler 是否在轮询期间调到 `dream_exit.propose()`：在 `propose()` 入口和每个 `return None` 前加 `logger.debug`，看它卡在哪个分支返回 None：
   - status 不是 REALITY_AFTERGLOW？→ 出梦没正确置态。
   - afterglow 未就绪？→ summary bg 没跑完/报错（看日志）。
   - 时机窗口/`_resolve_timing` 返回 None？→ `_AFTERGLOW_WAIT_HOURS=8.0` 等因素。
   - 被 gating 状态机/DND/活跃窗口挡？（`requires_state=[TriggerState.QUIET]`——如果你出梦后马上又在聊天，状态非 QUIET 会被压住）。
5. **最可能的两个根因**：(a) summary bg 异常导致 afterglow 没写；(b) 状态机门控（非 QUIET / 冷却 / 活跃窗口）一直压制。定位后小修即可，不要重写触发器。

### 交付
- 一段「根因 + 最小修复」说明 + 必要的小改（如补 except 日志、调 gating 条件）。若确认逻辑没问题只是触发条件苛刻，给出"如何稳定复现一次"的步骤即可。

## Part B · 软挽留（d-2，主开发）

### 目标（按用户拍板）
用户点 **wake（醒来）按钮** 退出梦境时：**若此刻梦境沉浸且情绪高涨**，先让叶瑄**软挽留一次**（说一句想留住对方的话），再让用户决定是否坚持醒来。不要求叶瑄有"显性挽留情绪"才触发——由**沉浸度 + 情绪强度**这两个客观量门控。**硬退口子保留**：用户坚持就一定能出。

### 现状（已核对）
- wake 按钮链路：`DreamWindow.tsx:169 handleWake → dreamExit()`（`shared/api/dream.ts:45`）→ Tauri `dream_exit`（`lib.rs:663`）→ `POST /dream/exit` → `force_exit_dream()`（**无条件硬退，当前点 wake 即静默立刻退出，毫无挽留**）。
- 状态机已有 `DreamStatus.DREAM_EXIT_REQUESTED`（`dream_state.py:22`）——**正好是"已请求退出、尚未关闭"的挽留窗口**，当前没被用上。
- `dream_turn()` 仅在 `status in (DREAM_ACTIVE, DREAM_CLOSING)` 才处理（`dream_pipeline.py:113`）——所以挽留期间若想继续对话，状态需能回到 DREAM_ACTIVE。
- 情绪/沉浸量可读：`dream_state.emotional_tension`（0–1，叶瑄梦内情绪张力）、`body.heat`（BodyState）、HUD 的 `dream_depth`。`lucid_mode`（`lucid_shared`/`non_lucid`）。

### 设计：新增 `/dream/wake` 作为"挽留闸门"，`/dream/exit` 保持纯硬退

> 关键：**不要改 `/dream/exit`**。它是保命的硬退。挽留只发生在新的 `/dream/wake` 里，且只拦一次。

#### 【后端】1. 挽留门控 + 端点 `admin/routers/dream.py` + `core/dream/dream_pipeline.py`

新增 `POST /dream/wake`（与 `/dream/exit` 并列），逻辑：
```
读 dream_state(uid)
若 status != DREAM_ACTIVE:
    # 不在梦里或已在关闭流程 → 直接走硬退兜底
    await force_exit_dream(uid); return {"retained": False, "exited": True}

gate = _should_retain(state)   # 见下
若 (not gate) 或 已挽留过本场梦(state.get("retention_offered_dream_id") == dream_id):
    await force_exit_dream(uid); return {"retained": False, "exited": True}

# 触发软挽留：置 EXIT_REQUESTED，标记本场已挽留，生成一句挽留台词
state["status"] = DREAM_EXIT_REQUESTED
state["retention_offered_dream_id"] = dream_id
save_state(...)
retention_text = await _generate_retention_line(uid, state)   # 见下，free-form
return {"retained": True, "retention_text": retention_text, "dream_id": dream_id}
```

`_should_retain(state)` 门控（阈值放模块常量，便于调）：
```
immersive = (dream_depth >= RETAIN_DEPTH_MIN)  # 或：入梦后有效轮数 >= N 且 lucid_mode != 某"快速"档
high_emotion = (emotional_tension >= RETAIN_TENSION_MIN) or (body.heat >= RETAIN_HEAT_MIN)
return immersive and high_emotion
```
- `dream_depth` 可复用 `dream_hud` 的派生（或简单用 emotional_tension + heat 估）。**沉浸**也可用「入梦后有效轮数 ≥ 3」近似，避免刚进梦就挽留。二选一，注释说明。
- 建议初始阈值（可后调）：`RETAIN_TENSION_MIN=0.55`、`RETAIN_HEAT_MIN=55`、`RETAIN_DEPTH_MIN=60` 或 轮数≥3。

`_generate_retention_line(uid, state)`（`dream_pipeline.py` 内，复用梦境 LLM 调用风格）：
- 走梦境 prompt 通道生成**一句**叶瑄想留住对方的话，高自由度、自然、不暴露机制、不写括号动作。指令骨架：
  ```
  （你察觉到她正要醒来离开这场梦。此刻梦里气氛还浓/情绪还热。
  你不想就这样让她走——说一句想留住她的话。怎么说由你此刻的状态决定：
  可以是轻声的挽留、半开玩笑的不舍、或一个"再待一会儿"的请求。
  只说一句，不要解释，不要括号动作。）
  ```
- 失败/超时 → 返回 None，调用方退化为直接硬退（fail-open 到退出，绝不卡住用户）。

#### 【后端】2. 留下/坚持 两个后续动作
- **留下**：`POST /dream/resume` —— 仅当 `status == DREAM_EXIT_REQUESTED` 时把 status 置回 `DREAM_ACTIVE`，返回 `{ok:true}`。让用户可以继续对话（`dream_turn` 重新可用）。其他状态调用是 no-op。
- **坚持醒来**：前端直接调既有 `POST /dream/exit`（`force_exit_dream`，无条件）。**不需要新后端**。

#### 【前端】3. wake 交互改造 `Emerald-client`
- `shared/api/dream.ts` + Tauri `lib.rs`：新增命令 `dream_wake`（POST `/dream/wake`）、`dream_resume`（POST `/dream/resume`），照既有 `dream_exit` 的命令+reqwest 写法。`backend allowlist`（lib.rs:1412 那张表）也要加这两条路由。
- `DreamWindow.tsx handleWake`：改为先调 `dreamWake()`：
  - `retained == false` → 和现在一样走退出（`dreamExit` 已在内部被后端调用并返回 exited，前端按原 wake 收尾流程关闭梦境窗）。
  - `retained == true` → **不关窗**：把 `retention_text` 作为叶瑄的一条梦境消息显示在 DreamChatPanel，并弹出两个选择：
    - 「留下」→ 调 `dreamResume()`，恢复对话（状态回 DREAM_ACTIVE）。
    - 「还是要醒来」→ 调 `dreamExit()`（硬退，必成功），收尾关窗。
  - UI 可复用现有 `DreamControlBar` / 一个轻量行内按钮组；不要做成模态死锁，**任何时候原有的强退入口都还在**。
- 注意：挽留只发生一次（后端 `retention_offered_dream_id` 去重），所以用户第二次点 wake 直接退出，不会被反复挽留。

### 验收
- 满足门控（聊到情绪高、沉浸）时点 wake → 出现叶瑄一句挽留 + 留下/醒来 选项。
- 点「留下」→ 能继续在梦里对话。
- 点「还是要醒来」或第二次点 wake → 立刻退出（硬退路径不变）。
- 不满足门控（刚进梦/情绪平淡）时点 wake → 和现在一样直接退出，无挽留。
- LLM 生成挽留失败 → 自动退化为直接退出，不卡住。
- `/dream/exit` 行为零变化（跑现有梦境相关测试确认）。
- 不变量 D 守卫：确认没有任何路径能让用户退不出梦。
